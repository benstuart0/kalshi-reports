Kalshi MM — Analytics Schema

All event types logged per trading session, with field-level documentation.

← Session reports

Every trading session emits structured events to parquet files. Each event type maps to one parquet file per session (e.g. fill_events.parquet, markout.parquet). The analytics overview report is generated by joining and aggregating these files.

Events share two common fields on every row: timestamp (UTC datetime of the event) and session_id (UUID identifying the run). Use these to join across tables or to filter to a specific session when files are concatenated.

SessionSummaryEvent session_summary.parquet
Written twice per session: once at boot (event_type = "session_start") and once at shutdown (event_type = "session_end"). The end row includes the full activity counters, final P&L, and a sanitized JSON snapshot of the config that was live during the run. Duration, fill counts, and P&L are zeroed on the start row.

event_typestr"session_start" or "session_end"

end_timestr (ISO)When stop() was called — equals timestamp on the end row
duration_secondsfloatWall-clock session length in seconds (0 on start row)
starting_balance_dollarsfloatPortfolio value at session open (cash + positions)
ending_balance_dollarsfloatPortfolio value at session close
ending_cash_dollarsfloatUnrestricted cash at close
ending_position_value_dollarsfloatMark-to-market value of open positions at close
session_pnl_dollarsfloatending_balance − starting_balance
total_fillsintTotal fill events received
total_orders_placedintTotal new orders sent to the exchange
total_cancellationsintTotal cancel requests issued
distinct_tickers_tradedintNumber of unique tickers that received at least one fill
exit_reasonstr"normal", "stop_loss", "signal", or "error"
read_requests_throttledintBlocking read-bucket rate-limit waits this session
read_try_acquire_failuresintNon-blocking read-bucket misses (request dropped, not queued)
read_total_wait_secondsfloatCumulative seconds lost to read rate-limit waits
write_requests_throttledintBlocking write-bucket rate-limit waits (place/amend/cancel)

FillEvent fill_events.parquet
One row per exchange fill. Captures the full market state at fill time, the inventory change, and the EV of the fill (microprice − fill_price − fee). Join with MarkoutEvent on fill_id to measure post-fill adverse selection.
FieldTypeDescription
tickerstrMarket ticker (e.g. SPX-26MAR06-T5900)
order_idstrExchange order ID that was filled
fill_id join keystrUnique fill ID from the exchange; used to join with MarkoutEvent
sidestr"yes" or "no" — which contract leg
actionstr"buy" or "sell"
fill_priceint (¢)Execution price in cents (1–99)
fill_quantityintNumber of contracts filled
microprice_at_fillfloat? (¢)OBI-weighted fair value at fill time; null if orderbook was one-sided
spread_at_fillint (¢)Best ask − best bid at fill time
best_bid_at_fillint (¢)Top-of-book bid at fill time
best_ask_at_fillint (¢)Top-of-book ask at fill time
inventory_beforeintNet position in this ticker before this fill (negative = short)
inventory_afterintNet position after this fill
estimated_ev_at_placementfloat? (¢)EV quoted when the order was placed — (edge − fee) × fill_prob. null if order record not found
time_since_placement_msint?Milliseconds the order rested before filling. null if unavailable
fill_ev_per_contractfloat? (¢)Realized EV: microprice_at_fill − fill_price − fee_per_contract. Positive = we bought below fair value (or sold above). null when microprice unavailable
fill_ev_totalfloat? (¢)fill_ev_per_contract × fill_quantity
realized_pnl_centsfloat (¢)Closed P&L if this fill closed an existing position; 0 for position-opening fills
MarkoutEvent markout.parquet
Adverse selection measurement. After every fill, microprice is sampled asynchronously at 6 horizons (1s, 5s, 10s, 30s, 60s, 300s), producing one row per horizon per fill — up to 6 rows per fill_id. A positive markout means the market moved in our favor after the fill; negative means we were picked off.

Formula: buy: microprice_at_horizon − fill_price  |  sell: fill_price − microprice_at_horizon. Positive = good for us.
FieldTypeDescription
fill_id join keystrJoins to FillEvent.fill_id
tickerstrMarket ticker
horizon_secondsintMeasurement horizon: 1, 5, 10, 30, 60, or 300
microprice_at_fillfloat (¢)Fair value captured synchronously at fill receipt time (baseline)
microprice_at_horizonfloat (¢)Fair value at the measurement horizon
markoutfloat (¢)Signed P&L vs fill_price at this horizon. Positive = still in the money
sidestr"yes" or "no"
actionstr"buy" or "sell"

AmendDecisionEvent amend_decisions.parquet
Logged every time a resting order is evaluated against a new candidate price — on every orderbook update for every active slot. The decision is KEEP (queue position advantage outweighs moving), REPLACE (EV of the new price exceeds EV of staying), or CANCEL (resting order is negative EV with no better alternative). This is the most granular view into the requoting logic.
FieldTypeDescription
tickerstrMarket ticker
sidestr"bid" or "ask"
decisionstr"KEEP", "REPLACE", or "CANCEL"
reasonstrFine-grained reason string from should_requote_side
current_priceint (¢)Price of the currently resting order
new_price optionalint? (¢)Candidate price computed from the latest orderbook; null if no +EV price exists
micropricefloat (¢)OBI-weighted fair value at decision time
reservation_price optionalfloat? (¢)Inventory-adjusted fair value (Avellaneda-Stoikov reservation price)
best_bidint (¢)Top-of-book bid
best_askint (¢)Top-of-book ask
spreadint (¢)best_ask − best_bid
ev_keepfloat (¢)Expected value of keeping the resting order (accounts for queue position advantage)
ev_movefloat (¢)Expected value of moving to the new price (back of queue)
ev_improvementfloat (¢)ev_move − ev_keep; positive means moving is better
fill_prob_keepfloatModel's fill probability estimate for the current resting order given its queue position
fill_prob_movefloatFill probability estimate if we move to the new price (back of queue)
contracts_ahead_currentintContracts ahead of our current order in the queue
contracts_ahead_newintContracts ahead if we move to the new price (typically depth at that level)
inventoryintCurrent net inventory in this ticker
FillWhileCancellingEvent fill_while_cancelling.parquet
A fill that arrived while a cancel or amend request was in-flight — the fill won the race to the exchange. The state machine handles this correctly; this event provides observability on how often races occur. Also fires on the amend path when a 404 response indicates the order was filled before the amend landed.
FieldTypeDescription
tickerstrMarket ticker
sidestr"bid" or "ask"
order_id optionalstr?Order ID involved in the race
fill_priceint (¢)Price at which the race fill occurred
fill_quantityintContracts filled in the race
cancel_in_flight_ms optionalfloat?Milliseconds the cancel/amend request had been in-flight when the fill arrived
inventory_before optionalint?Inventory before the race fill; null on the amend path
inventory_after optionalint?Inventory after the race fill; null on the amend path
request_typestr"cancel" or "amend"
Market & Position
FillProbabilityCalibrationEvent fill_probability_calibration.parquet
Sampled at a 1:200 rate inside estimate_fill_probability(). Each row captures the fill probability model's inputs and output at one call. Three contexts are logged: quote_sweep (initial quote calculation), keep_ev (EV of keeping a resting order), and move_ev (EV of moving to a new price). Join with fill_events on ticker + price + timestamp window for predicted-vs-actual calibration.

Note: because of the 1:200 sampling rate, a session with 1M+ rows in this table represents roughly 200M fill probability evaluations.
FieldTypeDescription
tickerstrMarket ticker
seriesstrSeries ticker (e.g. SPX-26MAR06)
sidestr"buy" or "sell"
contextstr"quote_sweep", "keep_ev", or "move_ev"
priceint (¢)Price level the estimate is for
contracts_aheadintContracts ahead in the queue at this price level
distance_from_touchint (¢)How far our price is from the best bid/ask
histogram_sourcestr"series", "fallback", or "none"
n_tradesintNumber of trades in the histogram backing this estimate
estimated_fill_probabilityfloatModel output: probability of filling in the expected window (0–1)
best_bidint (¢)Top-of-book bid at estimation time
best_askint (¢)Top-of-book ask at estimation time
spreadint (¢)best_ask − best_bid
microprice optionalfloat? (¢)OBI-weighted fair value at estimation time

CapitalExhaustionModeEvent capital_exhaustion_mode.parquet
Logged on each transition into or out of capital exhaustion mode. Entry is triggered either reactively (an InsufficientFundsError from the exchange) or proactively (cash drops below a configured threshold fraction of portfolio value). In exhaustion mode, only the side that reduces inventory is quoted. Exit is triggered when cash recovers to the configured recovery threshold.
FieldTypeDescription
modestr"on" (entering exhaustion) or "off" (exiting)
triggerstr"insufficient_funds" (reactive), "threshold" (proactive cash-below-limit), or "recovery" (exit)
cash_balancefloat ($)Cash available at transition time
position_valuefloat ($)Position mark-to-market at transition time
total_portfolio_valuefloat ($)cash_balance + position_value at transition
Infrastructure
NetworkLatencyEvent network_latency.parquet
One row per API call. Provides end-to-end latency plus a three-way breakdown: queue time (rate-limit wait + signing + serialization), wire time (HTTP round-trip), and parse time (JSON deserialization). The ob_to_placement_ms field measures the full path from orderbook update receipt to API call start — the key latency figure for market making competitiveness.
FieldTypeDescription
endpointstrE.g. "place_order", "amend_order", "cancel_order", "get_markets"
methodstrHTTP method: "GET", "POST", "DELETE", etc.
latency_msfloatTotal request time from start to response parsed (ms)
successboolTrue if HTTP 2xx
status_code optionalint?HTTP status code
ticker optionalstr?Ticker the request was for (where applicable)
order_id optionalstr?Order ID (for place/amend/cancel)
error_message optionalstr?Error message on failure
error_type optionalstr?"timeout", "network_error", or "http_error"
request_size_bytes optionalint?Request body size in bytes
response_size_bytes optionalint?Response body size in bytes
rate_limit_wait_ms optionalfloat?Portion of latency_ms spent waiting for rate limit tokens; null for non-blocking calls
queue_ms optionalfloat?Rate-limit wait + RSA signing + body serialization time
wire_ms optionalfloat?HTTP round-trip time (request sent → response headers received)
parse_ms optionalfloat?Response body read + JSON deserialization time
ob_to_placement_ms optionalfloat?Time from orderbook update receipt to this API call start. Only on place_order and amend_order
RateLimitEvent rate_limits.parquet
Logged whenever the rate limiter causes a blocking wait. The exchange enforces separate read and write token buckets; writes (place/amend/cancel) are non-blocking but this event fires when a blocking acquire on the read bucket has to wait.
FieldTypeDescription
bucketstr"read" or "write"
tokens_requestedfloatTokens consumed by this request
tokens_availablefloatTokens in the bucket before the wait
wait_time_secondsfloatHow long the request was blocked waiting for tokens
endpoint optionalstr?API endpoint being called
method optionalstr?HTTP method
bucket_utilizationfloat1 − (remaining_tokens / max_tokens) after the wait completes
